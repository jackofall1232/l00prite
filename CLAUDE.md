## 1. Mission

l00prite is a vendor-neutral protocol that gives AI coding agents durable, file-based
project memory. It scaffolds a project blueprint, a `.l00prite/` memory folder, and
Claude/Codex prompts into a target repo so any agent — Claude, Codex, GPT, Gemini, or a
future CLI agent — can pick up where another left off, using files in the repo instead of
vendor-specific session state.

Target user: developers who want AI agents to work on a codebase across multiple sessions
and multiple agent vendors without losing context, without silent unreviewed execution, and
without every handoff depending on one tool's proprietary memory.

This CLAUDE.md describes l00prite's own repo — the protocol and its scaffolding tools —
not a project generated by l00prite. It reflects what is actually built today, not a design
still in progress.

## 2. Architecture

l00prite has four protocol layers, all implemented as plain Markdown and JSON:

1. **Scaffold layer** — `.claude/commands/build-loop.md` (Claude Code slash command) and
   `.codex/prompts/build-loop.md` (Codex/CLI equivalent) ask clarifying questions, pick a
   complexity tier, and write a target project's `CLAUDE.md`, `.l00prite/` memory folder,
   `.codex/prompts/`, `.claude/prompts/`, and a tiered skeleton — then stop. Scaffolding
   never executes the generated project.
2. **Memory layer** — `templates/l00prite/` defines the `.l00prite/` folder shape copied
   into every scaffolded project: `blueprint.md`, `ledger.md`, `memory.md`, `constraints.md`,
   `failures.md`, `todos.md`, `heartbeat.json`, `state.json`, `lock.json`, `LOCKING.md`,
   `events/`, `reviews/`, `sessions/`. `examples/vendor-neutral-output/` is a filled
   reference copy of the same structure.
3. **Event layer** — pending interrupts (PR review comments, failed CI runs) are modeled as
   first-class JSON objects moved through `pending/ → processing/ → completed/` and handled
   via `.codex/prompts/event-loop.md` / `.claude/prompts/event-loop.md`
   (`Classify → Plan → Execute → Verify → Persist → Respond`), capped at one event per loop
   by default.
4. **Handoff layer** — `resume-loop.md`, `heartbeat.md`, `respond-to-review.md`, and
   `handoff-summary.md` (mirrored for both Claude and Codex, plus target-project template
   copies under `templates/codex/prompts/` and `templates/claude/prompts/`) let an agent
   resume, decide whether to continue, respond to reviews, and hand off, all through shared
   `.l00prite/` files.

Cutting across all four layers: `.l00prite/lock.json` plus `LOCKING.md` document a
cooperative lock/lease convention (check before writing, acquire if unlocked/released/
expired, respect an active unexpired lock owned by someone else, reclaim-and-log a stale
one, release before stopping) so agents working close together in time reduce — though do
not eliminate — the risk of racing writes to shared memory files.

No backend, no hosted service, no external dependencies. Everything is prompt files, JSON
schemas, and `scripts/validate-l00prite.js`, a Node script with no dependencies of its own.

### Not yet built: execution mode

l00prite's own `.l00prite/todos.md` names an **opt-in execution mode** as the next
milestone: a `--execute` flag on `build-loop`, a pre-flight display + explicit confirmation
gate before any execution loop starts, a standalone `execute-loop` prompt, and hard stop
conditions (max iterations, unfixable failing tests, any destructive operation, missing
secrets, ambiguous requirements, human review gates, and lock/lease conflicts). None of this
exists in the repo yet — no `execute-loop.md` file, no `--execute` flag handling, no new
`heartbeat.json`/`state.json` execution fields. This section intentionally does not claim
otherwise; it is a forward-looking design note for the next session, not a description of
current capability.

## 3. Requirements

- [x] Scaffold a project blueprint (`CLAUDE.md`) from a clarifying-questions conversation.
- [x] Scaffold a `.l00prite/` memory folder into the target repo, never overwriting
      existing files silently.
- [x] Provide Claude Code (`.claude/commands/`, `.claude/prompts/`) and Codex
      (`.codex/prompts/`) workflows with feature parity between the two.
- [x] Support resumable loops: a new session can read `.l00prite/` and continue the next
      smallest useful step.
- [x] Support heartbeat control (`heartbeat.json`) to decide whether a loop should continue,
      pause, or stop.
- [x] Model PR review comments and CI failures as first-class, trackable events with a
      documented lifecycle and ID format.
- [x] Provide a lock/lease convention so agents working close together in time do not
      silently corrupt shared memory files.
- [x] Provide a validator (`scripts/validate-l00prite.js`) that checks structural and
      protocol invariants across prompts, templates, and examples.
- [ ] Ship an opt-in execution mode with pre-flight confirmation and hard stop conditions
      (see "Not yet built" above) — next milestone, not yet started.

- [x] The repo has been dogfooded against its own protocol at least once (this file, and

Execution mode is explicitly **not** part of this release's Definition of Done — it is the

unmet item in this checklist.

## 5. Agent Operating Loop

- **Generator role** — writes or modifies one file per iteration, verifying each change
  before moving to the next. Does not invent requirements beyond what is stated in Section 3.
- **Evaluator role** — after each file is written or modified: runs
  `node scripts/validate-l00prite.js` when a validated file changed, checks JSON files parse,
  and rejects any change that silently alters the scaffold-only default behavior or the
  non-execution boundary.
- **Loop description** — generate one file → evaluator checks it → if it passes, update
  `.l00prite/ledger.md` and `.l00prite/todos.md` → move to the next file. If the evaluator
  rejects a change, fix it in the same iteration before moving on. Do not batch multiple
  unrelated files into one step.

## 6. Heartbeat Rules

- **Human review gates** — mandatory human review before: (1) any change to
  `.claude/commands/build-loop.md` (core scaffold command), (2) any change to
  `scripts/validate-l00prite.js` (validator logic), (3) declaring a release's Definition of
  Done met.
- **Branch policy** — feature work on a feature branch; no direct commits to `main`; each
  file change is its own commit describing what was written and what was verified; open a
  PR for maintainer review before merging.
